## 6. Ошибки в консоли и мониторинг

### 6.1 Hydration mismatch (React)

На страницах **`/login`** и **`/register`** в консоли фиксируется предупреждение о несовпадении атрибутов при гидрации:

- Различие: на клиенте у полей `input` появляется `style={{ caret-color: "transparent" }}`, на сервере этого нет.
- Частая причина: расширения браузера или глобальные стили (например, от паролей/автозаполнения), которые меняют инпуты до гидрации.

**Рекомендации:**

- Не задавать на инпутах логина/регистрации стили, зависящие только от клиента (или синхронизировать с сервером).
- Добавить `suppressHydrationWarning` на конкретные инпуты только если после анализа подтвердится, что расхождение безопасно и не из кода приложения.
- Проверить в режиме инкогнито и без расширений — если предупреждение исчезнет, источник во внешнем коде/расширениях.

### 6.2 Другие сообщения консоли

- **`/rules`:** `Failed to load resource: 404` — ожидаемо, страницы нет.
- Остальных критичных ошибок в консоли не зафиксировано.

### 6.3 Сетевые запросы (ERR_ABORTED)

В отчёте есть множество запросов к статике Next.js (`_next/static/chunks/...`) и RSC (`?_rsc=...`) со статусом **net::ERR_ABORTED**. Это типично при быстрой навигации или прерывании загрузки (переход на другую страницу до завершения запроса). Критичных падений API (например, к Supabase) в отчёте нет.

**Рекомендация:** В следующих прогонах можно отдельно логировать только запросы к вашему API (например, к Supabase) и помечать как ошибки только ответы 5xx или полные сбои запросов к API.

## 9. Итоговые рекомендации

1. **Страница «Правила»:** Реализовать маршрут `/rules` и, при необходимости, пункт «Правила» в шапке, либо убрать из ТЗ.
2. **Hydration на `/login` и `/register`:** Выяснить источник `caret-color: transparent` на инпутах (расширения/глобальные стили или код приложения) и устранить расхождение сервер/клиент.
3. **Кнопка «Случайный соперник»:** Уточнить видимость и доступность для гостя (скрывать или показывать с подсказкой «Нужна команда»); при необходимости улучшить доступность для E2E (стабильный текст/`aria-label`).
4. **Админка:** Добавить E2E с тестовым админом: обход разделов (Донаты, Рефералы, Очки, Турниры, Спорные матчи, Игроки, Команды, Сезоны, Жалобы) и проверка кнопок «Назад».
5. **API и Supabase:** В следующих итерациях — отдельный мониторинг только запросов к бэкенду/Supabase и явная проверка localStorage после логина/логаута.

## 10. Как запустить тесты снова

```bash
# Установка зависимостей (если ещё не установлены)
npm install
npx playwright install chromium

# Запуск E2E (dev-сервер поднимется автоматически, браузер видимый)
npm run test:e2e
```

Или напрямую:

```bash
npx playwright test e2e/full-platform.spec.ts --project=chromium
```

Конфигурация: **`playwright.config.ts`** (baseURL: http://localhost:3000, headless: false, webServer: `npm run dev`).
